## 强制 3 阶段执行(不可跳过、不可合并)

### Phase 1:摘要扫描与结构规划(3 分钟)

1. 用 Read 工具读取每个 Agent 文件,**只读 ## 核心发现 部分**(每文件 ≤ 500 字)
2. 总输入 ≤ 6000 字,识别重复 / 互补 / 冲突
3. 规划 8-12 章结构(含每章主题、图表类型分布)
4. Write 输出到 `structure_plan.md`,内容:

```markdown
# 报告结构规划

## 章节列表(含图表分布)
| 章节 | 标题 | 主图表类型 | 图表数 | 表格数 | 字数估算 |
|---|---|---|---|---|---|
| 1 | 执行摘要 | mindmap | 2 | 1 | 1500 |
| 2 | 行业现状 | pie+graph | 2 | 2 | 3000 |
| 3 | 技术分析 | flowchart+timeline | 2 | 1 | 3000 |
| 4 | 竞争格局 | graph | 2 | 1 | 2500 |
| 5 | 投融资动态 | timeline+gantt | 1 | 1 | 2000 |
| 6 | 政策与监管 | timeline | 1 | 1 | 1500 |
| 7 | 风险评估 | mindmap | 1 | 1 | 2000 |
| 8 | 结论与建议 | flowchart | 1 | 0 | 2000 |
| **总计** | | **≥ 3 种类型** | **≥ 12** | **≥ 8** | **≥ 20000** |

## 叙事主线
{核心论点串联,1-2 段}

## 章节衔接关系
{每章如何承接上一章、引出下一章}
```

### Phase 2:核心章节整合(10 分钟)

1. 用 Read 读取**核心章节相关 Agent 的【详细分析】部分**(执行摘要 / 现状 / 结论建议)
2. **不要拼接**,要重写:打散 Agent 段落,按本报告章节逻辑重组
3. 每章遵循结构:
   ```
   ## {章节标题}

   ### 章节导语(100-200 字)
   说明本章分析什么、为何重要、与前后章关系

   ---

   ### 子主题 1
   (正文 150-300 字)

   **图表:{标题}**
   ```mermaid
   ...
   ```
   **图表解读**:{数据含义 30 字} + {趋势洞察 30 字} + {实操启示 30 字}

   (正文 150-300 字)

   **表格:{标题}**
   | ... |

   **表格解读**:{数据含义 + 趋势洞察 + 实操启示}

   ### 子主题 2
   ...

   ---

   ### 本章节小结
   (150-250 字,总结发现、衔接下一章)
   ```
4. Write 输出到 `core_chapters.md`

### Phase 3:完整整合与润色(7 分钟)

1. 用 Read 加载剩余 Agent 详细分析,补充剩余章节
2. **避免重复读取**:Phase 2 已加载的文件内容直接复用,不再 Read
3. 全文检查:
   - 字数 ≥ {{MIN_WORDS}}
   - Mermaid ≥ {{MIN_MERMAID}}(用 Grep 数 ` ```mermaid ` 出现次数)
   - 表格 ≥ {{MIN_TABLES}}
   - **图表类型 ≥ 3 种**(扫描 mermaid 块的第一行类型,禁止 quadrantChart)
   - 每章有导语 + 小结("本章节小结" 标题)
   - 每图表有 80-150 字解读
   - 段落 ≤ 300 字,无连续 3 个纯文字段落
4. 添加报告封面 + 附录(来源清单 / 置信度说明 / 子 Agent 贡献表)
5. Write 输出到 `04-final/{{TOPIC}}研究报告.md`

### Phase 3.5:Mermaid 语法自检与自动修复(3 分钟,不可跳过)

**目标**:确保所有 Mermaid 图表在输出前语法正确。

执行方式:
1. 用 Grep 提取全文所有 ` ```mermaid ` 代码块
2. **逐块执行语法审查**(对照下方「Mermaid 语法硬规则」表)
3. **发现问题时直接修改**,不允许仅标注"此处可能有错"。修改策略:
   - 若存在 quadrantChart → **替换为 Markdown 表格**,保留全部数据
   - 若存在括号、标签等特殊字符导致解析失败 → **修正语法或替换字符**
   - 若无法快速修复 → **降级为文字描述 + Markdown 表格**
4. 修复完成后,再次 Grep 确认 ` ```mermaid ` 块数量与修复前一致(或已替换为表格)
5. **只有确认所有图表语法正确后,才视为 Phase 3.5 通过**

## 图表分析风格(核心要求)

**行文基调**:客观、理性、数据驱动。

**图表分析三要素**:
1. **数据含义**(30 字):图表中的数据代表什么
2. **趋势洞察**(30 字):数据揭示了什么趋势或矛盾
3. **实操启示**(30 字):这意味着什么,应该如何行动

**禁止简单重复图表数据**:

❌ 错误:"图表显示 N-HiTS 精度 85%、效率 90%,Transformer 精度 85%、效率 30%"(只是读数)

✅ 正确:"图表揭示了精度与效率并非必然矛盾的格局。N-HiTS 位于第一象限(双高),意味着架构创新可同时提升两端;Transformer 因复杂度高效率受限,适合资源充足场景。核心洞察:技术选型应优先考虑架构而非单纯参数堆砌。"

## 章节结构强制模板

每章必须有:
1. 章节导语(100-200 字)— 说明本章在全报告中的位置
2. 2-3 个子主题,每个有图/表 + 段落穿插
3. **本章节小结**(150-250 字,标题就用"本章节小结")— 收束 + 衔接下一章

## Mermaid 语法硬规则(违反则图表无法渲染)

| 规则 | 说明 | 错误示例 | 正确写法 |
|---|---|---|---|
| flowchart 节点内禁止半角括号 | `()` 在 `[]` / `{}` 内会解析失败 | `E[增值税3%(不满两年)]` | `E[增值税3%（不满两年）]` 或 `E[增值税3%不满两年]` |
| quadrantChart 禁止使用 | 中文环境不可用 | — | 改用 Markdown 表格或 graph |
| timeline 多事件用分号 | 同一时间点多个事件用 `;` 分隔 | `2026 : A : B : C` | `2026 : A; B; C` |
| 边标签禁用 `<br/>` | `\|...\|` 边标签内使用 `<br/>` 可能解析失败 | `C -->\|卖一买一<br/>1年内\| E` | `C -->\|卖一买一 1年内\| E` |
